# Planning and Task Breakdown

## Overview

Decompose work into small, verifiable tasks with explicit acceptance criteria. Good task breakdown is the difference between an agent that completes work reliably and one that produces a tangled mess. Every task should be small enough to implement, test, and verify in a single focused session.

In `mework`, the plan lives in the OpenSpec change: architectural decisions go in `openspec/changes/<name>/design.md`, and the task list goes in `tasks.md`. Both are generated by `/opsx:propose` and refined here; you tick `tasks.md` off during `/opsx:apply`.

## When to Use

- You have an OpenSpec proposal and need to break it into implementable units
- A task feels too large or vague to start
- Work needs to be parallelized across multiple agents or sessions
- You need to communicate scope to a human
- The implementation order isn't obvious

**When NOT to use:** Single-file changes with obvious scope, or when `/opsx:propose` already produced well-defined tasks.

## The Planning Process

### Step 1: Enter Plan Mode

Before writing any code, operate in read-only mode:

- Read the change's `proposal.md` and the relevant `openspec/specs/<capability>/spec.md` baseline
- Read the relevant `internal/...` packages and identify existing patterns and conventions
- Map dependencies between components
- Note risks and unknowns

**Do NOT write code during planning.** The output is `design.md` + `tasks.md`, not implementation. (`/opsx:explore` is the dedicated think-only mode if you need to investigate first.)

### Step 2: Identify the Dependency Graph

Map what depends on what. A mework change typically flows from schema/migration outward to the handler and write-back:

```
goose migration / schema (internal/store)
    │
    ├── provider adapter (internal/server/provider/<name>)
    │       │
    │       ├── webhook ParseTrigger + enqueue (internal/server/webhook)
    │       │       │
    │       │       └── job lifecycle + state machine (internal/server/jobs)
    │       │               │
    │       │               ├── daemon claim/run (internal/daemon, internal/agentrun)
    │       │               └── REST write-back via durable outbox
    │       │
    │       └── credential sealing (internal/server/secret)
    │
    └── end-to-end check (internal/integration TestFullPipelineE2E)
```

Implementation order follows the dependency graph bottom-up: build foundations first.

### Step 3: Slice Vertically

Instead of building all the schema, then all the handlers, then all the daemon wiring — build one complete path at a time:

**Bad (horizontal slicing):**
```
Task 1: Build entire schema
Task 2: Build all webhook handlers
Task 3: Build all job lifecycle code
Task 4: Connect everything
```

**Good (vertical slicing):**
```
Task 1: Webhook for provider X verifies signature and parses the @mework trigger
Task 2: A parsed trigger enqueues a deduped job (provider_code + external_event_id)
Task 3: Daemon claims the job, runs the AI CLI (prompt via stdin), acks done
Task 4: Server posts the result back over the provider REST API via the outbox
```

Each vertical slice delivers working, testable functionality and can be exercised by `TestFullPipelineE2E`.

### Step 4: Write Tasks

Each task in `tasks.md` follows this structure:

```markdown
## Task [N]: [Short descriptive title]

**Description:** One paragraph explaining what this task accomplishes.

**Acceptance criteria:**
- [ ] [Specific, testable condition]
- [ ] [Specific, testable condition]

**Verification:**
- [ ] Tests pass: `make test`   # go test -p 1 ./... (or: go test ./internal/server/jobs/...)
- [ ] Vet clean: `make vet`
- [ ] Build succeeds: `make build`  # or: go build ./...
- [ ] Manual check: [what to verify]

**Dependencies:** [Task numbers this depends on, or "None"]

**Files likely touched:**
- `internal/server/jobs/state.go`
- `internal/server/jobs/state_test.go`

**Estimated scope:** [Small: 1-2 files | Medium: 3-5 files | Large: 5+ files]
```

### Step 5: Order and Checkpoint

Arrange tasks so that:

1. Dependencies are satisfied (build foundation first)
2. Each task leaves the system in a working state (`make build` + `make vet` pass)
3. Verification checkpoints occur after every 2-3 tasks
4. High-risk tasks are early (fail fast — e.g. state-machine or dedup races)

Add explicit checkpoints:

```markdown
## Checkpoint: After Tasks 1-3
- [ ] `make vet` and `make test` pass
- [ ] `make build` produces bin/mework and bin/mework-server
- [ ] Core webhook → job → run flow works end-to-end
- [ ] Review with human before proceeding
```

## Task Sizing Guidelines

| Size | Files | Scope | Example |
|------|-------|-------|---------|
| **XS** | 1 | Single function or config change | Add a `ParseTrigger` workflow keyword |
| **S** | 1-2 | One handler or function | Add a new `/api/v1` endpoint |
| **M** | 3-5 | One feature slice | New provider adapter under `internal/server/provider/<name>/` |
| **L** | 5-8 | Multi-component feature | Sweeper requeue + heartbeat timeout handling |
| **XL** | 8+ | **Too large — break it down further** | — |

If a task is L or larger, break it into smaller tasks. An agent performs best on S and M tasks.

**When to break a task down further:**
- It would take more than one focused session (roughly 2+ hours of agent work)
- You cannot describe the acceptance criteria in 3 or fewer bullet points
- It touches two or more independent subsystems (e.g., webhook *and* secret sealing)
- You find yourself writing "and" in the task title (a sign it is two tasks)

## Plan Document (design.md) Template

Record decisions in the change's `design.md`; keep the actual task list in `tasks.md`.

```markdown
# Design: [Feature/Change Name]

## Overview
[One paragraph summary of what we're building and which capability it changes]

## Architecture Decisions
- [Key decision 1 and rationale — e.g. why REST write-back, not MCP]
- [Key decision 2 and rationale — e.g. reuse (provider_code, external_*_id), no migration]

## Dependency Graph
[Bottom-up ordering of internal/... packages]

## Risks and Mitigations
| Risk | Impact | Mitigation |
|------|--------|------------|
| Duplicate webhook delivery | Med | UNIQUE(provider_code, external_event_id) dedup |
| Dead runtime holding a job | High | Sweeper requeues claimed/running past heartbeat timeout |

## Open Questions
- [Question needing human input]
```

## Parallelization Opportunities

When multiple agents or sessions are available:

- **Safe to parallelize:** Independent feature slices, tests for already-implemented behavior, documentation
- **Must be sequential:** goose migrations, job state-machine changes, anything touching shared Postgres state (tests run `-p 1` for this reason)
- **Needs coordination:** Features that share the provider adapter interface or an API contract — define the contract first, then parallelize

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "I'll figure it out as I go" | That's how you end up with a tangled mess and rework. 10 minutes of planning saves hours. |
| "The tasks are obvious" | Write them in `tasks.md` anyway. Explicit tasks surface hidden dependencies and forgotten edge cases. |
| "Planning is overhead" | Planning is the task. Implementation without a plan is just typing. |
| "I can hold it all in my head" | Context windows are finite. `design.md`/`tasks.md` survive session boundaries and compaction. |

## Red Flags

- Starting implementation without a written `tasks.md`
- Tasks that say "implement the feature" without acceptance criteria
- No verification steps (`make vet`/`make test`) in a task
- All tasks are XL-sized
- No checkpoints between tasks
- Dependency order isn't considered (handler before its migration)

## Verification

Before starting implementation, confirm:

- [ ] Every task in `tasks.md` has acceptance criteria
- [ ] Every task has a verification step (`make test` / `make vet` / build / manual)
- [ ] Task dependencies are identified and ordered correctly
- [ ] No task touches more than ~5 files
- [ ] Checkpoints exist between major phases
- [ ] `design.md` records the architectural decisions and risks
- [ ] The human has reviewed and approved the plan

## mework notes

- The plan is OpenSpec: decisions go in `openspec/changes/<name>/design.md`, the
  ordered task list in `tasks.md`. `/opsx:propose` scaffolds both; you refine them
  here and tick `tasks.md` during `/opsx:apply`. This skill is the PLAN+TASKS half
  of `spec-driven-development`.
- Order tasks bottom-up along the pipeline (migration → adapter → webhook → jobs →
  daemon/agentrun → REST write-back), and let `internal/integration`'s
  `TestFullPipelineE2E` be the end-to-end checkpoint.
- Keep the invariants in acceptance criteria: prompts to AI CLIs over **stdin,
  never argv**; transactional job state machine with immutable terminal states;
  provider-agnostic schema keyed by `(provider_code, external_*_id)` (adding a
  provider needs no migration); AES-256-GCM-sealed credentials; `0600` files /
  `0700` dirs.
- Verification evidence for each task belongs under
  `openspec/changes/<name>/evidence/`. Tests are serialized: `make test` is
  `go test -p 1 ./...`, and DB-backed tests skip unless `TEST_DATABASE_URL` is set
  (`make test-db` starts Postgres).
